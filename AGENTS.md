# AGENTS.md

This document provides development rules and commands for AI agents and contributors working on Cometline.

## Repository Structure

This is a monorepo with three first-class modules tracked in a single git repository:

- **`comet-sdk/`** — Go library for provider-agnostic LLM I/O (streaming, retries, tool-call assembly, Anthropic/OpenAI/Codex adapters)
- **`cometmind/`** — Go agent runtime (agent loop, SQLite persistence, HTTP/SSE API, Discord gateway)
- **`cometline/`** — SvelteKit + Electron desktop shell (chat UI, settings, animations)

**Important:** There is no root `go.work` file. Run Go commands from `comet-sdk/` or `cometmind/`, not the repository root.

The historical standalone repos for these modules are archived. New development, issues, and pull requests all land in this repository.

The dependency direction is strictly one-way: `cometline` → `cometmind` → `comet-sdk`.

## Development Commands

### Root-level commands

```bash
make install          # Install frontend dependencies (pnpm install in cometline/)
make generate         # Regenerate OpenAPI clients (TS + Go) from openapi.yaml
make check            # Run all checks: codegen freshness, SDK tests, CometMind tests, Svelte checks
make build            # Build SDK, CometMind binary, and Cometline renderer
make package          # Build CometMind sidecar and package Electron app
make dev              # Build CometMind sidecar and launch Electron dev app
```

### Comet SDK (Go library)

```bash
cd comet-sdk
make test             # Run unit/integration tests (no live API calls, CI-safe)
make test-live        # Run live provider tests (requires API keys)
make build            # Verify compilation
make lint             # Run golangci-lint (requires golangci-lint installed)

# Run a specific test
go test -run TestStream_TextOnly ./provider/anthropic

# Live tests with specific provider
make test-live-anthropic
make test-live-openai
```

### CometMind (Go runtime)

```bash
cd cometmind
go test ./...         # Run all tests
go build ./...        # Verify compilation

# Run a specific test
go test -run TestPostMessageStreamsSSEAndPersistsUserTurn ./server

# Generate SQL code after schema changes
sqlc generate         # Requires sqlc installed
```

### Cometline (Desktop shell)

```bash
cd cometline
pnpm run dev          # Start Vite dev server + Electron
pnpm run build        # Production SvelteKit build
pnpm run check        # Type checking (svelte-kit sync && svelte-check)
pnpm run test         # Run Vitest tests
pnpm run lint         # Run ESLint
pnpm run generate:api # Regenerate TS client from openapi.yaml
```

## Generated Code

Several files are generated from source-of-truth definitions. **Do not edit generated files directly.**

### OpenAPI contract

- **Source:** `cometmind/openapi.yaml` (wire-contract source of truth)
- **TS client:** `cometline/src/lib/generated/cometmind-api/` (generated by `@hey-api/openapi-ts`)
- **Go types:** `cometmind/internal/apigen/types.gen.go` (generated by `oapi-codegen`)

After editing `openapi.yaml`:
```bash
make generate         # From repository root
# Or individually:
cd cometline && pnpm run generate:api
cd cometmind && go generate ./internal/apigen
```

### Database code

- **Schema:** `cometmind/internal/db/schema.sql`
- **Queries:** `cometmind/internal/db/queries/*.sql`
- **Generated:** `cometmind/internal/db/*.sql.go`, `db.go`, `models.go` (generated by sqlc)

After changing schema or queries:
```bash
cd cometmind && sqlc generate
```

**Migration note:** CometMind embeds `schema.sql` and tracks migrations with `schemaVersion` + `alterStatements` in `internal/db/migrate.go`. Schema changes for existing users need an incremental migration there, not just a schema edit.

### SSE event types

To add a new SSE event type:
1. Extend `StreamEvent` in `cometmind/openapi.yaml`
2. Add event type and emitter in `cometmind/internal/event/event.go`
3. Run `make generate`
4. Add reducer case in `cometline/src/lib/reducers/chat.ts`
5. Extend contract tests in `cometmind/internal/contract/contract_test.go`

## Runtime Environment

### CometMind

- **Settings:** `~/.cometmind/cometline-settings.json` (created by `go run . init` or Cometline; managed by Settings UI)
- **Legacy config:** `~/.cometmind/config.toml` (read only when JSON settings are missing)
- **Database:** `~/.cometmind/cometmind.db` (SQLite, pure Go via `modernc.org/sqlite`)
- **API:** `http://127.0.0.1:7700` (localhost only)
- **OpenAPI spec:** `cometmind/openapi.yaml`

**Environment variables:**
- `COMETMIND_PROVIDER`, `COMETMIND_BASE_URL`, `COMETMIND_API_KEY`, `COMETMIND_MODEL` — override config
- `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` — provider API keys (fallback to `COMETMIND_API_KEY`)

**CLI commands:**
```bash
go run . init                          # Create config/database, register workspace
go run . serve --port 7700             # Start HTTP/SSE API
go run . chat "message"                # Send a message (workspace-scoped)
go run . session list                  # List sessions
go run . gateway run --platform discord # Start Discord gateway
```

### Desktop settings

- **Location:** `~/.cometmind/cometline-settings.json`
- **Managed via:** Settings UI in Cometline
- **Contents:** Provider configs, active provider, appearance, shortcuts, app settings

Settings are synced to CometMind on save. If provider or runtime settings change, the sidecar restarts automatically.

## Key Features

### Default model

Users can set a default model in Settings → Providers → Model roles. This model is used for new sessions unless explicitly changed.

**Implementation:**
- `ProviderSettings` includes `defaultModelId` and `defaultProviderId`
- `modelStore.setProviders()` accepts default model params and selects it on startup
- `modelStore.selectDefault()` resets to default when returning to home page
- `+page.svelte` calls `selectDefault()` in `onMount`

### Slash commands

Built-in slash commands are defined in `cometline/src/lib/skills/slash-commands.ts`:
- `/change` — fork session into another workspace
- `/create-skill` — create a new Agent Skill
- `/model` — switch model for current session

The filter uses relevance scoring: name prefix match (3) > name substring (2) > description match (1).

### Agent Skills

Skills are reusable prompt templates stored in:
- `~/.cometmind/skills/{skill-name}/SKILL.md`
- `{workspace}/.agents/skills/{skill-name}/SKILL.md`
- `{workspace}/.claude/skills/{skill-name}/SKILL.md`

Invoke with `/{skill-name}` in the composer.

### ACP delegation

The `delegate_coding_task` tool spawns external coding agents (OpenCode, Claude Code, or any configured ACP agent) via ACP protocol.

Configure this under Settings → CometMind → ACP, persisted in `~/.cometmind/cometline-settings.json`.

### Discord gateway

Run CometMind as a Discord bot with the same agent runtime.

**Config:** Settings → CometMind → Discord, persisted in `~/.cometmind/cometline-settings.json`. Legacy `config.toml` is still accepted for migration.
```toml
[gateway.discord]
enabled = true
bot_token_env = "DISCORD_BOT_TOKEN"
workspace_path = "/path/to/project"
allowed_users = ["user_id"]
require_mention = true
```

**Start:** `cometmind gateway run --platform discord`

## Testing

### Test strategy

- **SDK tests:** Unit/integration tests without live API calls (CI-safe). Live tests behind `live` build tag.
- **CometMind tests:** Server tests use `httptest` and temporary SQLite databases.
- **Cometline tests:** Vitest for frontend logic, Svelte component tests.

### Live tests

SDK live tests require API keys and are skipped by normal `go test ./...`:

```bash
cd comet-sdk
go test -tags=live -timeout=60s ./...

# Or use Makefile targets
make test-live
make test-live-anthropic
make test-live-openai
```

**Environment variables for live tests:**
- `CUSTOM_API_KEY` — used first for both providers
- `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` — provider-specific keys
- `CUSTOM_BASE_URL` — override endpoint for both providers

### Fixtures

SDK stream tests use checked-in SSE fixtures under each provider's `fixtures/` directory. Update fixtures when parser behavior changes.

## Development Workflow

### Before starting work

1. Read `ARCHITECTURE.md` for system overview
2. Read `ARCHITECTURE_GUIDE.md` for detailed contributor map
3. Understand which module owns the feature you're changing

### Making changes

1. Identify the module that owns the change
2. Make changes in that module
3. If the change affects contracts (API, SSE, IPC), update all layers
4. Run `make check` to verify type safety and tests
5. Test manually with `make dev`

### Code style

- **Go:** Follow standard Go conventions. Use `gofmt` and `goimports`.
- **TypeScript/Svelte:** Follow existing patterns in the codebase. Use TypeScript strict mode.
- **Comments:** Add comments only when necessary to explain non-obvious logic.

### Commit messages

Use conventional commits:
```
type(scope): description

feat(cometline): add default model picker to settings
fix(cometmind): prevent tool execution path escape
refactor(comet-sdk): extract retry logic into helper
```

## Closed-Loop Self-Improvement

Cometline can improve itself using the same agent runtime:

1. Register this repo as the workspace (`cd cometmind && go run . init --workspace ..`, or open the repo in Cometline)
2. Ask CometMind to improve Cometline
3. CometMind calls `delegate_coding_task` to hand coding to OpenCode
4. Review test output in the parent session

**Verify command:** `cd cometmind && go test ./...`

## Common Tasks

### Add a new LLM provider

1. Implement `cometsdk.Provider` interface in `comet-sdk/provider/{name}/`
2. Add provider method constants/config mapping where needed in `cometmind/internal/config/` and `cometmind/internal/provider/factory.go`
3. Add provider defaults and validation in `cometline/src/lib/settings/schema.ts`
4. Add provider UI behavior in `cometline/src/lib/components/SettingsPanel.svelte` if the method needs custom fields
5. Update `ProviderMethod` types in `cometline/src/lib/types.ts` and `cometline/src/app.d.ts`

### Add a new built-in tool

1. Implement `cometmind/internal/tools.Tool` with `Spec() ToolSpec` and `Execute(ctx, input)`
2. Register it in `cometmind/internal/tools/registry.go`
3. Add unit tests for schema and execution behavior
4. Tools registered by the runtime are automatically exposed to the agent loop

### Add a new SSE event type

1. Extend `StreamEvent` in `cometmind/openapi.yaml`
2. Add event type to `cometmind/internal/event/event.go`
3. Add emitter function in same file
4. Run `make generate`
5. Add reducer case in `cometline/src/lib/reducers/chat.ts`
6. Update contract tests in `cometmind/internal/contract/`

### Add a new Agent Skill

1. Create `~/.cometmind/skills/{skill-name}/SKILL.md`
2. Write YAML frontmatter (name, description)
3. Write markdown body with trigger scenarios, workflow, examples
4. Skill is auto-discovered and invokable via `/{skill-name}`

## Troubleshooting

### Sidecar won't start

- Check `~/.cometmind/cometline.log` for sidecar errors
- Verify CometMind binary exists in `cometmind/dist/cometmind`
- Run `make port` to see if port 7700 is already in use

### Settings not persisting

- Check `~/.cometmind/cometline-settings.json` exists and is writable
- Verify JSON is valid (use `jq . ~/.cometmind/cometline-settings.json`)
- Check Electron console for IPC errors

### Tests failing

- Run `make check` to see all failures
- For SDK tests, ensure you're in `comet-sdk/` directory
- For live tests, verify API keys are set
